Global media delivery architecture

Build a relay fabric, not a complete graph.

A technical review of av-contrib, av-mesh, the cache/FEC layer, and the associated discovery, telemetry, and provisioning crates—benchmarked against the current low-latency media transport landscape.

Executive verdict Right media primitives.
Wrong global transport envelope.

The answer

Yes on the media architecture. No on the present network architecture.

The project has a credible nucleus: contribution protocols terminate at a regional edge; media is normalized into stream-addressed fMP4/CMAF-like artifacts; cache slots are format-agnostic; RaptorQ recovery is real; and the telemetry/qualification effort is unusually serious for this stage.

But the mechanism currently called the mesh is a small-cluster prototype. It gossips peers over unauthenticated UDP, sends locally produced slots to every known peer, does not forward remote slots, and broadcasts demand requests. That makes a publisher’s egress grow with every replica while providing no routed path through the “mesh.” It also lacks the congestion control, pacing, workload identity, authorization, durable desired state, and repair semantics required on the public Internet.

Strategic recommendation Preserve the media engine. Introduce a secure QUIC session substrate and reshape distribution into an interest-driven, bounded-degree, dual-parent relay DAG. Use reliable streams for control and backfill, datagrams for deadline-bound media, and RaptorQ as an adaptive repair tool inside that session—not as the session, topology, or security model.

Preserve / evolve / retire

The shortest decision log

Keep

The artifact boundary

Contribution adapters producing opaque, immutable media objects for a cache/relay layer is the right separation. Formalize object identity and lifecycle, but do not collapse protocol parsing into the core.

Keep, reposition

RaptorQ

Use it selectively when repair-by-retransmission would miss an object deadline. Drive parity from measured loss, RTT, deadline, and queue state; always retain a reliable fetch/backfill lane.

Keep

SRT / RIST / RTMP edge

They are useful contribution compatibility adapters. Add WHIP for browser/native WebRTC ingest and make protocol termination region-local.

Replace

Full-fanout UDP cache mesh

Replace the public-network path with authenticated QUIC sessions, explicit subscriptions, small peer degree, forwarding relays, paced queues, and reliable repair.

Evolve

LL-HLS

Keep it as the high-scale compatibility lane. Its part duration should reflect measured client RTT and Apple’s LL-HLS rules; 50 ms is not a sensible global default.

Do not promote

Prototype orchestration as authority

tcp-changes, VLAN broadcast discovery, and provider scripts can bootstrap labs. They should not become the authoritative global control plane.

Low-latency topology research

Choose the graph for the traffic pattern

Live media is one-to-many, deadline-sensitive, and heavily skewed by popularity. A complete peer graph spends origin bandwidth to avoid relay hops; a pure tree minimizes transmissions but is fragile. The production sweet spot is a measured, interest-driven relay DAG with two failure-independent parents for important streams.

Full mesh

Do not use globally. Direct paths, but peer state and links trend toward O(N²); source fanout is unbounded for a replicated stream.

Single tree

Efficient but brittle. O(N) object transmissions and bounded source load, but a parent or path failure cuts an entire subtree.

Recommended

Dual-parent DAG

Best core overlay. Small bounded degree, alternate parent, relay caching, path diversity, and interest-driven construction.

DHT / rendezvous

Use for lookup, not media paths. Consistent hashing can assign stream anchors/catalog shards; latency-oblivious overlays are poor forwarding routes.

Anycast + edge

Use for client entry. Anycast/GeoDNS/content steering finds an edge; it does not replace the relay overlay or per-stream placement.

N − 1minimum transmissions per object to reach N nodes in any tree
≈ 2Nedges for a sparse dual-parent overlay, versus N(N−1)/2 in a complete graph
3–6recommended steady-state peer degree per relay, adjusted by role and capacity

The degree target is a design recommendation to validate, not an Internet standard. Select parents using active RTT/loss/jitter/goodput, provider/ASN/region diversity, cost, capacity, and object-deadline performance—not latitude/longitude alone.

Gap assessment

The issues that define readiness

These are ordered by ability to cause compromise, congestion collapse, silent corruption, or a failed global rollout—not by implementation convenience.

P0

The mesh wire protocol has no security boundary

Peer hello, cache objects, replica requests, node identity, and discovery are accepted over UDP without cryptographic peer identity, confidentiality, authorization, or replay protection. CRC protects against accidental bit errors, not malicious modification. The HTTP control routes can provision infrastructure, drain nodes, and warm streams without an authentication guard; contribution ingest surfaces likewise lack token enforcement and quotas.

Evidence: playlists/src/mesh.rs peer/packet handling; av-mesh/src/main.rs control routes; av-contrib/src/bin/av-contrib.rs ingest routes; discovery/src/vlan.rs four-byte IP announcements.

P0

Raw UDP sends have no congestion controller or pacer

The sender encodes an object, then emits its source and repair packets in a loop for each peer. There is no congestion window, pacing clock, acknowledged delivery signal, deadline queue, or fairness response. On real Internet paths, bursts create the loss that FEC then tries to repair, while competing unfairly with other traffic. QUIC datagrams explicitly solve the missing session/security/congestion envelope while retaining unreliable delivery.

Evidence: RaptorQ send loops in the current cache and contribution forwarders; compare RFC 9221 and RFC 9002.

P0

Full fanout is labeled as a mesh but does not route

Every local slot is sent to every peer; remotely received slots are marked replicated and are not forwarded. Demand requests are broadcast. The source must therefore be directly connected to every selected replica. Gossip carries a capped peer list every 500 ms, and sync scans streams every 20 ms. This is reasonable for a lab cluster and fundamentally wrong for hundreds of global nodes.

Evidence: FRAME_MAGIC = PLMESH1, hello/sync intervals, peer cap, request_replica, and replication flags in playlists/src/mesh.rs.

P0

There is no authoritative, durable control loop

Placement decisions are computed from transient self-reported telemetry and acted on by broadcast requests. There is no durable desired-state record, generation number, lease/fencing model, idempotent command status, rollback, or leader election. The Linode crate performs useful imperative operations, but provisioning, DNS, private-IP selection, and application readiness are not one reconciled transaction.

Evidence: av-mesh/src/replication.rs, tcp-changes in-memory channels, and linode/src/lib.rs. These coordination semantics should live in a small native controller; they do not require a container orchestrator.

P1

RaptorQ is overextended from repair mechanism to transport strategy

RaptorQ is an excellent object-level erasure code. It does not supply sessions, identity, congestion response, routing, prioritization, or reliable recovery after a deadline prediction is wrong. The codebase already contains adaptive policy, scheduling, and backfill ideas; the main hot paths currently use a fixed small repair policy instead. Integrate those richer primitives behind per-path feedback.

RaptorQ is specified for object delivery in RFC 6330. The recommendation to combine it with QUIC is an architectural inference from the two protocols’ complementary properties.

P1

Placement is geographic, not network-aware

Latitude/longitude, continent, static capacity, and demand are a useful bootstrap. They are not enough to optimize tail latency or resilience. The scheduler needs active path telemetry, delivery-deadline misses, egress cost, provider/ASN/failure-domain diversity, hysteresis, and penalties for route churn. An edge’s nearest region is not necessarily its best Internet path.

Evidence: geographic scoring in av-mesh/src/replication.rs. ALTO Path Vector (RFC 9275) is a useful model for representing measured path/cost properties; it need not be adopted literally.

P1

LL-HLS tuning is too aggressive for a global default

A 50 ms part target increases request and scheduling overhead and is below the RTT of many global clients. Apple requires Part Target to be at least the 95th-percentile round-trip time, recommends at least three times that RTT, and recommends one second. Keep a lower experimental profile for controlled paths, but choose a standards-compliant measured default for public playback.

Source: Apple HLS Authoring Specification. Also audit blocking reload, delta updates, preload hints, rendition reports, and hold-back.

P1

Ingress needs product-grade admission and a broader object model

av-contrib is promising but currently centers on H.264/AAC and listener-configured streams. It needs authenticated stream admission, tenant quotas, bounded bodies, rate limits, publisher failover, timestamp provenance, discontinuity handling, codec configuration epochs, and an ABR/keyframe-aligned ladder. Add WHIP as the standards-track WebRTC ingest surface.

WHIP is RFC 9725 and includes bearer-token authorization behavior. WHEP remains an expired Internet-Draft, so treat it as an experimental egress compatibility lane.

P2

Process boundaries and supervision need hardening

The mesh main module combines data plane, edge serving, metrics, operator UI, control, provisioning, multiple transports, and format logic. Spawned mesh tasks are not consistently surfaced to a supervisor. Split the system into crates/process roles with explicit health, restart, bounded queues, cancellation, and overload behavior. A playback bug must not be able to provision or drain infrastructure.

Evidence: av-mesh/src/main.rs (~10.6k lines) and av-contrib/src/bin/av-contrib.rs (~4.9k lines) in the reviewed snapshot.

2026 standards position

Use stable foundations; isolate moving targets

The current state of the art is converging on QUIC as the secure congestion-controlled substrate, object-oriented media semantics above it, and relays as first-class participants. The application protocol layer is still moving, so interoperability belongs behind versioned adapters.

RaptorQ decision

RaptorQ is a good choice—with a precise job description

This review does not recommend removing RaptorQ. It recommends protecting its value by using it as deadline-aware object repair inside a complete transport system. The local implementation, recovery tests, adaptive policy, scheduling, and backfill work are strategic assets.

Where it wins

Repair before a retransmission can arrive

  • A media object has a known playout deadline and the remaining time is near or below a path RTT.
  • Bursty or reordered loss makes a small amount of proactive repair cheaper than waiting for a NACK/ARQ round trip.
  • The same object feeds multiple receivers or caches, so encoding work can be amortized and extra repair symbols can be generated as needed.
  • Objects are naturally bounded—CMAF chunks, access units, or MOQT-like objects—with explicit identity and dependency.
  • A relay can combine initial source symbols, modest proactive parity, and on-demand additional symbols without regenerating the media.
Where it loses

When parity replaces transport fundamentals

  • On a clean, low-RTT path, fixed proactive parity wastes bandwidth compared with a cheap reliable repair.
  • If packets are emitted in an unpaced burst, the sender can create its own loss; adding parity can make that congestion worse.
  • A receiver may need enough symbols to decode a block, adding CPU, memory, and completion delay for very small/urgent objects.
  • One fixed repair ratio cannot fit peers with different RTT, loss bursts, queues, bandwidth, and deadlines.
  • FEC cannot authenticate a sender, hide content, select a route, enforce fairness, or guarantee recovery after the object’s deadline.

The rule RaptorQ overhead is part of the congestion budget, not an exemption from it. Let measured loss, remaining object deadline, repair RTT, queue state, and object importance choose the number of symbols. Keep reliable fetch as the correctness path. This makes RaptorQ a competitive advantage rather than a source of uncontrolled overhead.

North-star architecture

A secure, object-native relay fabric

One canonical object graph, three delivery lanes, and a control plane that describes desired state rather than broadcasting imperative hope.

Below the data plane: a separate desired-state controller, node agents, service registry, placement engine, path measurement, and durable audit/event log.

Data plane

Every relay session has two lanes

  • Deadline lane: QUIC datagrams carrying object fragments, paced and congestion-controlled; optional RaptorQ parity chosen per object/path.
  • Certainty lane: QUIC streams for subscriptions, catalogs, control messages, initialization objects, and object FETCH/backfill.
  • Priority: init/config and current keyframe group outrank audio, current delta frames, parity, then stale backfill.
  • Overload: shed expired/stale objects before growing queues. Never let a delayed frame make newer frames late.
Control plane

Desired state, identity, and measurement

  • A durable controller owns node membership, stream placement, relay parent sets, epochs, drain state, and rollout generation.
  • Node agents reconcile versioned desired state and report observed state with command IDs, ACKs, and failure reasons.
  • Relays run as single-purpose services on ordinary VMs under the host service manager. No Kubernetes control plane is required.
  • SPIFFE/SPIRE-style short-lived workload identities authenticate mTLS; per-role authorization limits publish, subscribe, control, and provision rights.
  • Placement consumes active path probes and observed object delivery, plus provider/ASN/zone diversity, capacity, and egress cost.

Three product latency lanes

Treat latency as an explicit product contract. Exact SLOs must be established by multi-region measurement; these are target bands for engineering and capacity planning.

Canonical key

tenant / stream / track / epoch / group / object / version / hash

Immutable, idempotent writes. Explicit init objects, discontinuities, deadlines, dependencies, and codec configuration epochs.

Parent selection

Score measured p95 RTT, loss bursts, jitter, goodput, deadline misses, load, cost, and failure-domain diversity. Add hysteresis and minimum tenure to avoid route flapping.

Join and repair

Subscribe at a decodable group boundary, fetch init/keyframe reliably, receive live objects via datagrams, and request precise missing objects before their usefulness deadline.

Path forward

From strong prototype to global fabric

This sequence minimizes throwaway work. Security and measurement come first because every topology and FEC optimization depends on trusted path feedback and honest end-to-end latency data.

Now → 30 days

Make the system safe and measurable

  • Threat-model all ingress, peer, playback, metrics, dashboard, control, and provisioning surfaces. Default-deny public exposure.
  • Put the current UDP mesh on a private encrypted network immediately; add node allowlists while QUIC identity is built.
  • Add authentication/authorization, bounded request bodies, quotas, timeouts, and rate limits. Remove unauthenticated provisioning/drain/warm actions.
  • Define a versioned canonical object envelope and end-to-end capture/ingest/package/relay/edge/player timestamps.
  • Split transport, cache, object model, relay, node-agent, and edge-serving interfaces into crates without changing behavior.
  • Run the existing two-/three-region qualification and a seven-day soak; record baseline latency, loss, bandwidth, parity, and recovery cost.

  • No unbounded network allocation
  • Traceable glass-to-glass latency
  • Seven-day regional baseline published

  • Add per-peer pacing, bounded priority queues, object deadlines, PMTU handling, qlog, ACK/loss feedback, and overload shedding.

Definition of greatness

Measure outcomes, not packet cleverness

Release-blocking invariants

  • No object or command is accepted without an authenticated principal and authorized role.
  • No unbounded queue, frame, object, body, peer list, or in-flight repair set is reachable from the network.
  • No stream can make origin egress grow linearly with the number of playback edges.
  • No repair strategy can bypass congestion control on the public Internet.
  • No node placement or drain command can execute twice with a different result.
  • No stale media object may delay a newer decodable group.

Proposed SLO scorecard

  • Availability and successful start by delivery lane, region, ASN, device, and codec.
  • Glass-to-glass p50/p95/p99, plus capture→ingress, package, relay, edge, and player-buffer stages.
  • Deadline hit rate, stale-object discard rate, repair success before deadline, parity efficiency, and goodput.
  • Rebuffer ratio, startup time, live-edge distance, rendition switch time, and A/V sync error.
  • Parent churn, failover interruption, route stretch, relay degree, origin fanout, and control reconciliation lag.
  • CPU, memory, bandwidth, and egress cost per contributed and delivered media minute.
Global rollout stop condition Do not expand the current UDP gossip/fanout protocol onto the public Internet. Expansion should wait until peer identity, encryption, authorization, pacing/congestion control, bounded queues, explicit subscriptions, and reliable object repair are present in the same end-to-end path.

Method and limits

Review scope

This is an architecture and code-path review, not a penetration test, formal protocol proof, production capacity test, or compliance certification. It covered av-contrib, av-mesh, playlists, raptor-fec, tcp-changes, discovery, linode, message-packetizer, operational scripts/status notes, and current primary standards sources. The working trees contained active uncommitted changes and were not modified except for this report.

Before committing to numeric SLOs, run synchronized multi-region tests on representative access networks and publish raw distributions. Before public launch, perform a focused threat model, parser fuzzing campaign, dependency review, external security assessment, and abuse/cost-amplification test.
